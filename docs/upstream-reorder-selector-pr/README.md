# Upstream contribution: add `reorder` support to HA's `select` selector

## Status: diffs prepared, not yet forked/pushed/opened (by design — see below)

## Why this exists

This project's `wifi_geolocation` integration wants users to drag-reorder
which geolocation backend(s) to try, in priority order. Research (during
this session, using two Explore agents against the real current source of
both `home-assistant/core` and `home-assistant/frontend`) found:

- HA's **frontend** (`ha-selector-select.ts`) has fully implemented
  drag-to-reorder for the `select` selector's `multiple: true` case since
  **PR #18180** (merged 2023-10-24) — `ha-sortable`/SortableJS wiring, drag
  handles, the works. It's already used internally by several Lovelace
  card-feature editors (`customizable-list-feature.ts`,
  `hui-select-options-card-feature-editor.ts`,
  `hui-alarm-modes-card-feature-editor.ts`), which build the selector
  schema directly in TypeScript, bypassing the gap described below.
- The **Python backend** (`homeassistant/helpers/selector.py`,
  `home-assistant/core`) never added a `reorder` key to
  `SelectSelectorConfig`/`SelectSelector.CONFIG_SCHEMA`, unlike
  `AreaSelectorConfig`/`EntitySelectorConfig`, which both already support
  it via a shared `_validate_selector_reorder_config` validator. Since
  `make_selector_config_schema` defaults to `PREVENT_EXTRA`, passing
  `reorder: true` to a `select` selector today raises "extra keys not
  allowed" *before* it ever reaches the already-working frontend code.
- No open or merged issue/PR anywhere (`home-assistant/core`,
  `home-assistant/frontend`, `home-assistant/architecture`) proposes fixing
  this specifically — genuinely unclaimed, and unusually low-risk to fix
  since the hard part (frontend) is already shipped and production-proven.

**This project also ships a local runtime workaround** for the gap
(`custom_components/wifi_geolocation/_selector_patch.py`) so real
drag-to-reorder works today, without waiting on this PR — see that file's
docstring. This upstream contribution is the "do it properly, for
everyone" follow-up; the local patch can be deleted once this ships in a
stable HA release.

## What's in this directory

Real diffs, generated by actually cloning `home-assistant/core`
(`homeassistant/helpers/selector.py`, `tests/helpers/test_selector.py`)
and `home-assistant/home-assistant.io`
(`source/_docs/blueprint/selectors.markdown`) at their current `dev` HEAD,
making the edit, and running `git diff` — not hand-approximated. Both
Python files were confirmed to still parse correctly in isolation after
the edit (note: `home-assistant/core`'s live `dev` branch currently has an
unrelated, pre-existing syntax error elsewhere in `selector.py` — a
Python-2-style `except vol.Invalid, vol.MultipleInvalid:` in a different,
unrelated selector class — which blocks compiling the *whole* file right
now regardless of this change; not something this PR needs to fix).

- **`core-selector.patch`** — `homeassistant/helpers/selector.py`: adds
  `reorder: bool` to `SelectSelectorConfig`, wraps `SelectSelector.
  CONFIG_SCHEMA` in `vol.All(..., _validate_selector_reorder_config)`
  (previously a bare `make_selector_config_schema({...})` call), reusing
  the exact same validator function `AreaSelector`/`EntitySelector`
  already use. No `__call__` changes — confirmed unnecessary by their
  precedent (`reorder` is a pure pass-through flag never read by Python
  validation logic; it exists only so the frontend receives it).
- **`core-tests.patch`** — `tests/helpers/test_selector.py`: adds one
  `multiple: True, reorder: True` valid case to
  `test_select_selector_schema`'s parametrize list, and two invalid cases
  (`reorder` without `multiple`) to the existing
  `test_select_selector_schema_error`.
- **`docs-selectors.patch`** — `home-assistant.io`'s
  `source/_docs/blueprint/selectors.markdown`: adds the `reorder` entry to
  the "Select selector" section, wording matched to the existing
  area/entity `reorder` doc entries.

This exact schema-construction pattern (add `reorder` field + wrap in
`vol.All` + reuse `_validate_selector_reorder_config`) is also
independently verified working via this project's own test suite —
`tests/custom_components/test_wifi_geolocation_selector_patch.py`, 4
passing tests — since the local patch module constructs the identical
shape.
